# v6.0.0a36

## Predictive factories — Midpoints, Secondary Progressions, Solar Arc

This alpha completes the core Western-astrology predictive toolkit.
Together with the already-shipped `PrimaryDirectionsFactory`, kerykeion
now exposes the four classical techniques every professional western
astrologer expects:

| Technique | Factory | Calculation root |
|---|---|---|
| Primary Directions | `PrimaryDirectionsFactory` (since 6.0.0a30) | Diurnal rotation, Placidus semi-arc |
| Secondary Progressions | `SecondaryProgressionFactory` (this release) | "Day-for-a-year" symbolism |
| Solar Arc Directions | `SolarArcFactory` (this release) | Progressed Sun arc applied uniformly |
| Midpoints | `MidpointFactory` (this release) | Pure geometry on natal points |

## What's new

### `MidpointFactory`

```python
from kerykeion import AstrologicalSubjectFactory, MidpointFactory

natal = AstrologicalSubjectFactory.from_birth_data(
    "John", 1990, 6, 15, 14, 30,
    lng=12.5, lat=41.9, tz_str="Europe/Rome", online=False,
)

mids = MidpointFactory.compute(natal, aspect_orb=1.0)
for m in mids:
    print(f"{m.point_a}/{m.point_b}: {m.midpoint_position:.2f}° {m.midpoint_sign}"
          f"  (90° dial: {m.midpoint_modulus_90:.2f}°)")
    for a in m.aspects_to_midpoint:
        print(f"   activated by {a.point_name} ({a.aspect}, orb {a.orb:.2f}°)")
```

- Computes every unordered pair of active points (default 16 → 120 pairs).
- Each `MidpointModel` carries the **shorter-arc midpoint** longitude,
  the sign + position-within-sign, and the **90° dial position**
  (`longitude % 90`) that cosmobiology / Uranian practitioners read by
  default.
- Optional **aspect-to-midpoint** detection: for every midpoint we
  scan the active points for a configured aspect within `aspect_orb`
  degrees. The result lists the activating points so the consumer can
  surface "midpoint pictures" (`A = B/C` notation) directly.

### `SecondaryProgressionFactory`

```python
from kerykeion import SecondaryProgressionFactory

progressed = SecondaryProgressionFactory.compute(
    natal,
    target_iso_utc_datetime="2026-04-25T00:00:00Z",
)
print(progressed.sun.sign, progressed.moon.sign)
```

- Maps real-time onto the progressed scale via the canonical formula
  `progressed_days = real_years` (with `real_years = (target − birth) /
  365.25`), reusing the natal location so the same houses / angles can
  be compared.
- Returns a regular `AstrologicalSubjectModel` — every existing tool
  (aspects, dignities, chart drawer) keeps working unchanged.
- All natal settings (zodiac type, sidereal mode, house system,
  perspective, altitude, active points, timezone) are propagated.
- **BCE support**: natal subjects with `year < 1` (astronomical
  numbering) and BCE target years/ISO timestamps are fully supported
  via Julian Day arithmetic, bypassing Python `datetime` limitations.
- **Custom ayanamsa (USER sidereal mode)**: `custom_ayanamsa_t0` and
  `custom_ayanamsa_ayan_t0` are persisted on the model and propagated
  to the progressed chart.

### `SolarArcFactory`

```python
from kerykeion import SolarArcFactory

arc = SolarArcFactory.compute(
    natal,
    target_iso_utc_datetime="2026-04-25T00:00:00Z",
)
print(f"Solar arc: {arc.solar_arc:.4f}°")
for hit in arc.directed_to_natal_aspects:
    print(f"  {hit.directed_point} {hit.aspect} natal {hit.natal_point}"
          f" (orb {hit.orb:.2f}°)")
```

- Computes the progressed Sun via `SecondaryProgressionFactory`, takes
  the **forward zodiacal difference** with the natal Sun, applies that arc
  to every requested natal point, and returns a structured
  `SolarArcSubjectModel`. BCE natal subjects are supported.
- Each directed point carries its natal and directed positions, the
  natal/directed signs, and a `sign_changed` flag — the sign-ingress
  signal that Brennan, Ebertin, Hand all flag as primary.
- Optional **directed-to-natal aspect detection** with configurable
  orb produces the actionable timing list (the directed Mars
  conjunction natal Saturn → year of constructive struggle, etc.).
  Natal targets use the subject's own `active_points`, so extra points
  (Vertex, asteroids, Arabic parts) are checked automatically.

## Why now

Phase 2 of the consumer roadmap (Astrologer Studio v3) needs all three
calculations available before the UI can ship the predictive chart
pages. By landing the factories here, in `kerykeion` proper, both
`astrologer-api` and any third-party consumer can reuse them without
duplicating ephemeris work.

## Backward compatibility

Strictly additive:

- No existing class, method, or attribute is removed, renamed, or
  semantically changed.
- `custom_ayanamsa_t0` and `custom_ayanamsa_ayan_t0` added to
  `AstrologicalBaseModel` as `Optional[float] = None` — no impact on
  existing models. A Pydantic `model_validator` enforces pair integrity
  (both set or both `None`), and requires both when `sidereal_mode` is
  `"USER"`.
- `DOUBLE_CHART_TYPES` centralized in `charts_utils.py` and imported
  by `draw_planets.py` and `chart_data_factory.py` (was duplicated).
- SVG layer extended: a new `"Progression"` chart type is rendered by
  `ChartDrawer` via a dedicated `ProgressionChartRenderer`. Sizing logic
  was refactored into renderer hooks (`is_dual_wheel()`, `get_initial_width()`,
  etc.), which may cause minor layout differences in auto-sized dual-wheel
  charts. Consumers with SVG golden files or screenshot tests should review
  the baselines and verify existing baselines still match.
- 5 new SVG golden-file baselines added: 4 Progression (CE) + 1
  Progression (BCE).

### `"Progression"` chart type

The `ChartType` literal now includes `"Progression"`. Since
`SecondaryProgressionFactory.compute()` already returns a standard
`AstrologicalSubjectModel`, it slots directly into the dual-chart
pipeline:

```python
from pathlib import Path

natal = AstrologicalSubjectFactory.from_birth_data(...)
progressed = SecondaryProgressionFactory.compute(natal, target_year=2026)
data = ChartDataFactory.create_progression_chart_data(natal, progressed)
drawer = ChartDrawer(data, theme="classic")
drawer.save_svg(output_path=Path("charts_output"), filename="progression")
```

Updated files: `kr_literals.py`, `kr_models.py`, `chart_data_factory.py`,
`charts_utils.py`, `chart_drawer.py`. Progression uses the Transit
biwheel layout (natal inner ring, progressed outer ring).

### Context serializer for predictive models

Two new converter functions in `context_serializer.py`:

- `solar_arc_to_context(SolarArcSubjectModel)` → XML with directed
  points and directed-to-natal aspects.
- `midpoints_to_context(list[MidpointModel])` → XML with midpoint
  table, 90° modulus, and activation data.

Both are callable via `to_context()`.

Note: `SecondaryProgressionFactory` returns `AstrologicalSubjectModel`,
which `to_context()` already supports — no special handling needed.

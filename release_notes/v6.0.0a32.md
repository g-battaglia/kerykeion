# v6.0.0a32

## Glyph-center metadata on every ChartPoint

This alpha adds two attributes — `kr:cx` and `kr:cy` — to every
`<g kr:node="ChartPoint">` emitted in the chart SVG. Their values are the
exact glyph-center coordinates in chart SVG root coords, already computed
by the drawing code before it writes the markup. The change is purely
additive: no existing attribute is removed, renamed, or semantically
changed, and the rendered geometry is byte-identical across all styles.

## Why

Frontends that layer interactivity on top of kerykeion's SVG (tooltips,
click-to-focus, hit-testing) need the rendered glyph center to pick the
right symbol when padded hit-areas overlap in dense clusters — a very
common situation in tight stelliums and at the angles.

Measuring the center from the DOM is unreliable:

- Our `<symbol>` definitions omit `viewBox` by design. As a result
  `getBoundingClientRect` and `getBBox` on a `<use>` referencing one of
  our symbols return 0×0 (Firefox, WebKit) or an implementation-defined
  value (Blink). Frontends that depend on those measurements observe
  sporadic misalignment between cursor position and glyph selection.
- Parsing the `<g>` transform chain works for the `modern` style but
  diverges from `classic`, which wraps the symbol with
  `translate(-12 -12)` and places `<use x=X y=Y>` directly under the
  ChartPoint group. A consumer that wants to support both styles has to
  maintain two separate code paths.
- Each symbol has its own internal transform (`Sun` uses
  `translate(1, 4)`, `Mars` uses `translate(2, 0)`, etc.), so there is
  no single hardcoded offset that maps `<use>` attributes to the visual
  center uniformly across all planets and angles.

Emitting the coordinates directly removes every one of these problems.
The consumer reads two attributes, applies the root SVG's
`getScreenCTM()`, and obtains the exact viewport-space glyph center in
two lines of code.

## Usage

```javascript
// Given any ChartPoint <g> element in the injected SVG:
const cx = parseFloat(chartPoint.getAttribute('kr:cx'))
const cy = parseFloat(chartPoint.getAttribute('kr:cy'))
const svg = chartPoint.ownerSVGElement
const ctm = svg.getScreenCTM()
const screenPt = new DOMPoint(cx, cy).matrixTransform(ctm)
// screenPt.x / screenPt.y is the viewport-pixel glyph center.
```

## Coverage

Both SVG-emitting paths in `draw_planets.py` are covered:

- `_generate_point_svg` — single natal/composite/return charts, both
  `classic` and `modern` styles route through this function.
- Inline transit-chart path (around line 1069) — emits the transit
  planet symbols with an outer `<g>` wrapper and its own scale factor;
  the glyph center is the same `(point_x, point_y)` in root coords.

## Backward compatibility

Additive only. Existing consumers that ignore the new attributes keep
working with zero changes. The test suite (including SVG snapshot
fixtures) verifies that all other attributes, elements, and structural
ordering are unchanged.
